SECTION 1 · BACKEND LOGIN CODE (2 of 4)

Server bootstrap, config, data clients, middleware

src/server.ts

Fastify bootstrap. Configures pino logging with redaction rules for authorization, cookie, password, code, refresh_token — these never appear in logs. Registers plugins: sensible, helmet, cors, rate-limit (100 req/min default). Decorates Fastify with app.auth so routes use {preHandler: app.auth}. Mounts 4 route plugins (health, auth, otp, me). Global error handler. Graceful shutdown on SIGINT/SIGTERM closes pg pool + Redis client.

src/config/env.ts

Single Zod schema validates every env variable at boot. If DATABASE_URL is missing, the app crashes at boot with a precise error, not at the first request. Exports a typed env: Env object and isProd helper.

src/db/pg.ts

Postgres pool via pg. max: 10 deliberately small — PgBouncer-ready: a fleet of N app instances behind PgBouncer doesn't exhaust RDS max_connections=200. SSL on in production. Exports pool, query(), pingDb(), closeDb().

src/db/redis.ts

ioredis client with keyPrefix (so multiple environments can share one Redis). Supports rediss:// for TLS to ElastiCache. Key conventions:

otp:{channel}:{id} SHA256 hash, TTL 600s fail:{userId} incrementing counter lock:{userId} presence = locked sess:{userId} ZSET scored by createdAt otprl:{userId} OTP rate-limit counter

src/middleware/auth.ts

Bearer-token guard for protected routes. Three checks:

  1. Header has Bearer ...
  2. JWT verifies under RS256 public key (issuer + audience)
  3. Session still active in Redis — logout/revoke/quota-eviction takes effect immediately, before JWT expires

On success attaches req.user: AccessClaims with {sub, sid, role, org}.

MODULE 1 · foundations/ · LIVE on AWS

KMS keys + IAM roles

The security floor. Everything else encrypts with these keys and assumes these roles.

kms.tf

3 customer-managed KMS keys, annual rotation enabled. Each key policy: root account full control, scoped service principals (RDS, ElastiCache, S3, CloudTrail, etc.), aws:SourceAccount condition to block cross-account misuse.

  • alias/vidyouth-prod-data — RDS, ElastiCache, EBS, app S3
  • alias/vidyouth-prod-secrets — every Secrets Manager entry
  • alias/vidyouth-prod-audit — CloudTrail + S3 audit archive

iam.tf

4 IAM roles + 1 instance profile.

  • ec2-role — AmazonSSMManagedInstanceCore + CloudWatchAgentServerPolicy + AmazonEC2ContainerRegistryReadOnly + inline policy scoping secretsmanager:GetSecretValue to vidyouth-prod/* and kms:Decrypt to the data+secrets keys
  • rds-monitoring-role — AmazonRDSEnhancedMonitoringRole
  • flow-logs-role — inline policy for VPC Flow Logs → CloudWatch
  • events-role — trust policy for events.amazonaws.com (placeholder)

Instance profile: vidyouth-prod-ec2-profile wraps ec2-role for the launch template.

variables.tf

Inputs: name_prefix, account_id (from root data.aws_caller_identity), tags (merged into every resource).

outputs.tf

Feeds downstream modules:

  • data_kms_arn → data, observability, application
  • secrets_kms_arn → secrets module
  • audit_kms_arn → observability module
  • ec2_role_arn + ec2_instance_profile_name → application
  • rds_monitoring_role_arn → data
  • flow_logs_role_arn → networking

Why this matters

  • Encryption — every other module accepts a KMS ARN as input; no resource gets created with default service-managed keys
  • Least privilege — even a compromised EC2 instance can only fetch vidyouth-prod/* secrets
  • No static AWS credentials on EC2 — instance profile + IAM role grants temporary tokens via IMDSv2

Why auto-generate

No human ever sees the master DB password in plain text. Terraform generates → Secrets Manager stores → KMS encrypts → IAM scoping ensures only EC2 role can read. Rotation = one terraform apply.

Why recovery_window_in_days = 7

If you accidentally terraform destroy, secrets enter 7-day pending-deletion. Restore with aws secretsmanager restore-secret. Fat-finger protection.
